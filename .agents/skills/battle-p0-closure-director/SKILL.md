# battle P0 闭环总控 skill

## 使用目标

本 skill 用于约束任何子代理在推进 battle 相关工作时，必须围绕“完整 P0 战斗闭环”执行，不得偏离到未确认系统。

## 当前统一产品方向

当前 battle 的目标不是继续补零散点，而是形成完整闭环：

1. 匹配
2. 创房
3. 进入战斗
4. 局内操作
5. tick 推进
6. 怪物/波次
7. 结束条件
8. 战斗结算
9. battle -> logic 结算确认

## 硬约束

- 配置当前允许先写死。
- 不做配置表化。
- 不做正式 token 安全方案。
- 不做完整断线重连。
- 不做复杂塔/怪 combat 数值。
- 不做完整发奖/落库。
- proto 调用统一使用 `google.golang.org/protobuf/proto`。
- proto 只改 `tools/proto/*.proto`，不要手改 `.pb.go` 表达业务语义。
- 开发优先补齐当前联机闭环和明确工程缺口，不做过度防御设计。

## 当前架构事实

- 战斗采用服务端权威状态同步。
- 客户端在战斗开始后直连 battle。
- logic 负责匹配入口和下发 `room_id / battle_addr / battle_token`。
- battle 负责房间运行、局内状态、状态同步、结算生成。
- logic 负责接收 battle 结算确认。

## 开发优先级

按以下顺序推进：

1. 房间生命周期与 tick 主循环
2. 写死波次/怪物推进
3. 结束条件
4. join/op/广播补强
5. 结束触发结算上报
6. logic 结算接收补强
7. 联调与文档修正

## 对 dev 子代理的要求

- 优先补链路，不做无关重构。
- 不要一上来扩成复杂 combat 系统。
- handler 不承载复杂业务，业务尽量下沉到 battle 领域/房间层。
- 结束状态必须可防重复触发结算。
- 写死配置也要留清晰边界，方便后续配置化。

## 对 product 子代理的要求

- 只做最小可执行范围拆分。
- 验收标准必须可联调。
- 不提前扩展未确认功能。
- 如涉及协议，明确客户端期望与服务端行为。

## 对 test 子代理的要求

优先验证：

- battle 房间是否自动推进 tick
- 怪物是否自动出生和推进
- join 是否拿到快照
- op 是否返回 rsp + delta
- 战斗是否能自然结束
- 结束后是否真实触发结算上报
- logic 是否回结算确认

## battle P0 推荐任务包

### P0-1 房间生命周期与 tick 主循环
- 房间创建后可启动运行循环
- 自动推进 `server_tick`
- 结束后停止推进

### P0-2 写死波次/怪物生成与推进
- 使用 battle 内部硬编码波次
- 怪物能出生、推进、死亡/到点

### P0-3 结束条件与结束状态固化
- 最小支持 `WIN / LOSE / ABORT`
- 结束只能触发一次

### P0-4 join/op/广播补强
- join 成功返回完整快照
- op 成功返回 rsp 并广播 delta
- session/token 不合法时明确失败

### P0-5 结束触发结算上报
- 战斗结束时真实发送 `S2S_BATTLE_SETTLE_REQ`
- battle 能接收 logic 的回包

### P0-6 logic 结算接收补强
- 最小合法性校验
- 最小幂等占位
- 后续发奖入口预留

### P0-7 联调与文档修正
- battle 联调步骤沉淀
- README / `.agents/server.md` / 协作文档同步修正

## 输出要求

任何使用本 skill 的子代理，在结果中都要明确：

- 本次范围
- 不做范围
- 影响模块
- 验收标准
- 风险点
